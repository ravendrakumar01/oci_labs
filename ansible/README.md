# Ansible — post-provision configuration

Terraform provisions infra (VM + network + disk). Ansible configures the VM
over SSH. IPs come from a **dynamic inventory** (fetched live from OCI), and the
SSH key is **generated by Terraform** per VM.

---

## One-time setup (Windows → WSL Ubuntu)

1. Install Ansible + OCI SDK:
   ```bash
   sudo apt update && sudo apt install -y ansible python3-pip
   pip install oci --break-system-packages
   ```

2. Install collections:
   ```bash
   cd /mnt/d/MY-DATA/STUDY/TerraForm/Terraform_learn/oci_labs/ansible
   ansible-galaxy collection install -r requirements.yml
   ```

3. Create OCI config for the inventory plugin — `~/.oci/config`:
   ```ini
   [DEFAULT]
   user=<OCI_USER_OCID>
   fingerprint=<FINGERPRINT>
   tenancy=<TENANCY_OCID>
   region=ap-mumbai-1
   key_file=~/.oci/oci_api_key.pem
   ```
   Copy your OCI API private key into WSL:
   ```bash
   mkdir -p ~/.oci
   cp /mnt/c/Users/DBA/.oci/oci_api_key.pem ~/.oci/oci_api_key.pem
   chmod 600 ~/.oci/oci_api_key.pem ~/.oci/config
   ```
   (These are the same values that are in `terraform.tfvars`.)

4. Set the ANSIBLE_CONFIG env var (folder is on a Windows drive, so Ansible
   ignores ansible.cfg unless pointed to explicitly):
   ```bash
   echo 'export ANSIBLE_CONFIG=/mnt/d/MY-DATA/STUDY/TerraForm/Terraform_learn/oci_labs/ansible/ansible.cfg' >> ~/.bashrc
   source ~/.bashrc
   ```

## Fill in the DEV compartment OCID (one time)

Edit `inventory/oci.yml` → set `compartment_ocid`. Get it from Windows:
```powershell
terraform -chdir=envs/dev output -raw compartment_id
```

---

## Get the Terraform-generated SSH private key

Terraform generates the VM keypair and stores the private key in state.
Extract it to a file for Ansible/SSH (run on Windows where Terraform is):
```powershell
terraform -chdir=envs/dev output -json instance_private_keys | `
  ConvertFrom-Json | Select-Object -ExpandProperty 'dev-app-01' | `
  Out-File -Encoding ascii $HOME\.ssh\dev-app-01
```
Then in WSL, copy it and lock permissions:
```bash
cp /mnt/c/Users/DBA/.ssh/dev-app-01 ~/.ssh/dev-app-01
chmod 600 ~/.ssh/dev-app-01
```
(This path matches `private_key_file` in ansible.cfg.)

## Run

```bash
cd /mnt/d/MY-DATA/STUDY/TerraForm/Terraform_learn/oci_labs/ansible

# STEP 1: See what OCI returns (live hosts/IPs)
ansible-inventory --graph

# STEP 2: Connectivity test
ansible all -m ping

# STEP 3: Baseline — system update + mount attached volume

# All servers:
ansible-playbook playbooks/site.yml

# Only ONE specific VM (e.g. the one just provisioned):
ansible-playbook playbooks/site.yml --limit dev-app-01
```

## What the baseline playbook does

`playbooks/site.yml` runs on any server and:
1. Updates all system packages (`yum update -y` equivalent)
2. Mounts an attached block volume (`/dev/sdb` -> `/data`) — only if one is attached

It is idempotent — safe to re-run. New servers get updated/mounted; already-done
servers stay unchanged. Use `--limit <vm-display-name>` to target just one VM.

## Verify
- Browser: `http://<LB-IP>` → nginx page
- On VM: `df -h /data` → 50GB mounted

## Notes
- **Dynamic inventory** = no manual IPs; if a VM is recreated (new IP), just
  re-run — Ansible picks up the new IP automatically.
- **Per-VM keys**: each VM with `generate_ssh_key = true` gets its own keypair.
  For multiple VMs, extract each key and set `ansible_ssh_private_key_file` in
  host_vars per host.
- Static fallback: `ansible-playbook -i inventory/hosts.ini playbooks/site.yml`
